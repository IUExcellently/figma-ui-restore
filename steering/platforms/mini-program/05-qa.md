# mini-program · 小程序视觉验收

小程序验收不默认使用 Web PC 的 chrome-devtools 尺寸规则。

## 优先验收方式

如果项目支持，应优先使用：

- 微信开发者工具
- 支付宝 / 抖音等对应小程序开发者工具
- uni-app / Taro 的小程序预览方式
- 项目已有真机或模拟器预览流程

不可用时，降级为结构化自检，并明确说明未完成真实小程序预览验收。

## 建议检查尺寸

- 320
- 375
- 390
- 414
- 430

如项目有目标机型列表，以项目目标机型优先。

## 检查重点

- 目标平台、技术栈和设计稿宽度 `dw` 已确认。
- width / height / line-height 已按小程序白名单规则转换。
- `font-size` 未使用宽高 rpx 换算公式。
- `padding` / `margin` 未使用宽高 rpx 换算公式。
- 状态栏、胶囊、navbar、tabBar 边界已确认。
- 自定义导航栏避让胶囊和状态栏。
- 底部 fixed 区域避让 safe-area bottom。
- 本地背景图使用 `image` 绝对定位复刻，而不是 WXSS 本地 `background-image`。
- 图片 `mode` 与 Figma 语义一致。
- Figma 临时资源未进入生产。
- 页面滚动和局部 `scroll-view` 边界清晰。
- 原生组件层级没有遮挡异常。
- 长文本截断、换行和完整展示方案正确。


## 小程序补充验收

完成小程序 UI 还原后，必须补充以下验收项：

- 微信开发者工具或对应小程序开发者工具预览通过。
- 至少检查 iOS 和 Android 各一台真机；如果无法真机检查，必须明确说明未执行。
- 检查首屏加载是否出现图片闪烁、背景图错位、布局跳动。
- 检查底部 fixed 区域是否遮挡内容。
- 检查自定义 navbar 是否避开胶囊区域。
- 检查 safe-area-bottom 是否正确处理。
- 检查页面返回、tabBar 切换、分享入口是否正常。
- 检查列表为空、加载中、加载失败、接口慢响应状态。
- 检查长文本、省略、多行展示。
- 检查包体分析结果：
  - 主包是否变大。
  - 当前页面资源是否误入主包。
  - 是否有重复图片资源。
- 不可用真机 / 开发者工具时，必须降级说明，不得声称完成真实验收。

## 自检报告补充

小程序自检报告必须说明：

- 运行时方言：微信原生 / uni-app / Taro React / Taro Vue / 其他。
- 新增资源数量、位置和包体影响。
- 是否存在开放能力、授权、支付、分享、订阅消息。
- 是否存在原生组件层级覆盖风险。
- 是否执行真机检查；未执行时说明原因。
