# common · 跨平台 UI 还原基础规则

本文件只放对小程序、H5、Web PC 都成立的规则。任何平台专属规则都放入 `steering/platforms/<platform>/`。

## 开发前确认

写代码前必须先输出开发前确认报告，并等待用户确认。报告至少包含：

- 目标平台与判断依据
- 输入材料类型：单尺寸 / 多尺寸 / 多状态 / 多页面
- 页面壳或 Layout 边界
- 本次生成范围
- 项目技术栈和可复用组件扫描结果
- 设计 token 与颜色处理方案
- 资源处理方案
- 组件拆分方案
- ViewModel 草案
- 语义化事件设计
- 适配 / 响应式策略
- 风险点与待确认事项

## Figma 事实优先

Figma 事实来源包括：

- Figma MCP 原始数据
- 截图真实渲染结果
- `meta.json`
- `reference.tsx`
- 用户补充说明

若来源冲突，以截图真实渲染和 Figma 原始数据为优先，并在开发前确认报告中标记风险。

## reference.tsx className 解析

如果 `reference.tsx` 中的 `className` 来自 Tailwind、UnoCSS、CSS Modules、自定义原子类或 Figma 生成类名，必须先解析成 CSS 声明，再按目标平台和项目样式方案落地。除非目标项目本身使用同一套 class 框架，否则不得原样照搬。

## 字体与颜色

- 不在组件内写 `font-family`。
- 字重映射：Regular = 400，Medium = 500，Semibold / Bold = 600。
- 颜色类样式以 Figma 实际色值为最高依据。
- 只有项目已有变量与 Figma 实际色值完全一致时才复用。
- 语义相近但色值不同的变量不得强行复用。
- 少量局部颜色可直接写在当前 class；重复或页面级核心颜色再沉淀为变量。
- 新增颜色变量使用 `@figma-` 前缀，不使用节点 ID、图层名、组件名命名。

## 资源与图标

- Figma MCP 临时资源 URL 不得进入生产代码。
- 图片、图标、装饰资源优先复用项目已有资源；无法复用时按平台规则处理。
- 图标尺寸优先取语义图标容器或直接包裹层的宽高 / size 信息，不使用内部 Vector / img 的 inset 切片尺寸。
- 装饰资源不参与主内容流布局，不承载交互。

## 组件职责与事件

页面或页面容器负责：

- 接口请求
- 路由跳转
- 权限判断
- 数据组装
- 业务编排
- ViewModel 转换

展示组件负责：

- 接收 props / ViewModel
- 渲染 UI
- 局部交互状态
- emit / 触发语义化事件

组件不得只暴露裸 `click` / `tap` / `dblclick`，应暴露业务语义事件，例如：

```text
view-detail
view-all
select-item
submit-form
create-invite
refresh-list
```

## 布局基础

- 按业务模块拆分，不按 Figma 图层机械拆分。
- 组件之间外部间距由父级页面或父级布局容器控制。
- 子组件不默认携带外部 margin。
- 组件内部不能无脑使用 gap；gap 只用于同质元素等距排列。
- 横向布局必须识别固定区、弹性区和剩余区。
- 重要长文本需要溢出策略和完整内容展示方案。

## 验收基础

开发完成后必须输出结构化自检报告。真实截图或预览验收方式由平台规则决定；无法执行真实验收时，必须明确说明降级，不得声称已经完成截图验收。
