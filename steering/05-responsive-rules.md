# 05 · 响应式与断点规则

## 单尺寸 / 多尺寸输入判断

多尺寸源数据是增强输入，不是必需输入；本规则不删除、不弱化、不重写原有单尺寸还原逻辑。

处理前先分流：

- 单页面单尺寸 UI：沿用现有单稿还原流程，按业务模块输出通用响应式推导策略。
- 单页面多尺寸 UI：启用多尺寸基准适配流程。
- 同尺寸多状态 UI：归类为状态变体，不进入响应式断点差异。
- 多页面多尺寸 UI：先按业务页面分组，再对每个页面单独判断。

多页面分组必须结合 pageName、frameName、nodeId、目录名、截图命名、`meta.json` source 信息判断是否属于同一业务页面，不能只按尺寸分组。

`metadata.xml` 是可选参考输入；如果存在，则纳入结构判断。不能要求每个尺寸都必须存在 `metadata.xml`。

## 多尺寸源数据扫描

当本地目录存在多个尺寸稿时，先扫描并记录：

- 尺寸来源：目录名、文件名、`meta.json` frame width / height、截图尺寸。
- 是否有 Sidebar / Menu / Header。
- Sidebar 宽度。
- content 区域宽度。
- content inner width。
- 是否为状态变体，例如 empty / loading / error。
- 是否有 `reference.tsx`、`meta.json`、截图、可选 `metadata.xml`。

如果文件名无法识别尺寸，应从 `meta.json` 的 frame width / height、截图尺寸、目录名中推断；仍无法确认时列为待确认项。

## 多尺寸基准选择

单页面多尺寸 UI 必须以最大宽度的默认有数据态为基准开发。

其他尺寸只作为模块级响应式参考，用于校准布局、间距、显隐、换行、组件 variant、内容宽度和溢出策略。

如果最大宽度只有 empty / loading / error 状态，必须列为待确认项，不得直接作为默认基准稿。

同尺寸下的 empty / loading / error 是状态变体；状态变体只能用于补充状态 UI，不得当作响应式断点。

多尺寸里的样例数据差异不能当成响应式规则。例如某个 item 在 375 的文案和 1440 不一致，只能视为 mock 数据差异，不能写成媒体查询逻辑。

不同尺寸中同一装饰图或图标的 Figma 临时 URL 不一致时，不要直接当作不同生产资源；优先沉淀为同一项目本地资源，并通过 CSS 尺寸、位置、显隐策略适配。

## 响应式优先级

响应式判断优先级如下：

1. 用户明确说明优先。
2. 全局 Layout 策略跟随项目现状。
3. content 区域优先参考多尺寸 Figma。
4. 单尺寸输入时，才使用通用响应式推导规则。

如果多尺寸稿缺少 520，520 只作为工程验收断点，根据 768 和 375 的模块变化进行插值推导，不得创造与两侧尺寸冲突的新布局。

## Sidebar / Menu / Header 归属

Sidebar / Menu / Header 属于全局 Layout 策略。

如果项目已有 Layout，默认跟随项目现有 Layout，不得因为 Figma 某个尺寸隐藏了菜单就擅自修改 Layout。content 区域根据真实可用宽度适配。

如果多尺寸稿中 Sidebar / Menu / Header 在不同尺寸下不一致，必须列为开发前待确认项：

- 该差异由全局 Layout 处理，还是本次页面处理。
- 768 及以下菜单是否隐藏。
- 隐藏后是否变为抽屉、顶部入口或完全不展示。
- 页面适配基于 viewport，还是基于 Layout 后的 content 可用宽度。

## 全局断点

```less
@screen-xxl: 1440px; // Figma 基准稿 / 大屏桌面
@screen-lg: 1024px;  // 窄桌面 / 横向平板
@screen-md: 768px;   // iPad / 平板竖屏临界
@screen-sm: 520px;   // 手机布局策略临界
@screen-xs: 375px;   // 最小设计保障宽度
```

## 断点含义

- 1440：设计基准，不一定每次都写媒体查询。
- 1024：常规桌面 / 窄桌面。
- 768：平板 / 后台窄内容区。
- 520：进入手机布局策略。
- 375：项目最小设计保障宽度。

375 以下不作为主要适配目标，允许降级或横向滚动。

## 核心原则

响应式不是全局一刀切。

每个业务模块必须单独声明：

- 1440 怎么排
- 1024 怎么排
- 768 怎么排
- 520 怎么排
- 375 怎么排
- 是否允许换行
- 哪些元素固定
- 哪些元素弹性
- 哪些文本省略

## 后台页面优先看 content 宽度

后台页面有 Sidebar / Header，viewport 不等于 content 可用宽度。

如果项目支持 container query，优先基于内容容器判断。

如果不支持，再使用项目统一媒体查询变量。

## 示例策略

### 欢迎模块

- 始终左右布局。
- 左侧上下布局。
- 右侧不收缩。
- 不换行。
- 左侧长文本 ellipsis。

### 快捷入口卡片

- 大屏横向排列。
- 宽度不足时换行或纵向排列。
- 每个卡片可 `flex: 1 1 360px`。

### 预警 + 统计卡

- 大屏：预警主区域 + 右侧统计卡一行。
- 中屏：统计卡整体换到第二行，两个卡均分。
- 小屏：全部纵向。
